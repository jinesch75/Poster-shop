admin: persist backfill report via cookie, not URL query string

The previous implementation stuffed every per-poster result row into the
post-action redirect URL as `?report=<encoded>`. That worked fine for
small runs but 503'd at production scale: a 21-poster re-stamp produced
a redirect URL ~1.5 KB long, and Railway's edge timed out (or rejected
the request) before Next.js could render the result page. The action
itself completed successfully — only the result-table view was broken.

This commit moves the report payload off the URL and into a short-lived
HTTP cookie. The flow is now:

  1. Action finishes → set `maintenance_report` cookie (httpOnly,
     SameSite=strict, path-scoped to /admin/maintenance/backfill, 5-min
     max-age) carrying the same encoded summary string the URL used to.
  2. Action issues a clean `redirect('/admin/maintenance/backfill')`.
  3. Page Server Component reads the cookie via `cookies()` from
     next/headers, parses, renders the existing results table.

No behavioural changes for the user: identical result table, identical
status counts, identical per-row layout. The cookie expires on its own
after 5 minutes, so a stale browser tab eventually returns to the
"no last run" state without intervention. Re-running an action within
5 minutes overwrites the cookie with the new run's results.

Implementation details:

- Cookie value is `encodeURIComponent(<the same encoded summary string
  we used in the URL>)`. URL-encoding before storage keeps the cookie
  parser happy on commas / pipes / spaces inside per-poster detail
  strings (e.g. error messages from try/catch).
- `path: '/admin/maintenance/backfill'` scopes the cookie tightly so
  it isn't echoed on every admin request.
- `httpOnly: true` keeps it out of any client-side JS (the page is a
  Server Component and reads the cookie server-side, so JS access is
  unnecessary).
- New helpers `encodeReport(results)` and `persistReport(results)`
  centralise the cookie-set logic so both `runBackfill` and `runRestamp`
  call it identically.
- `BackfillPage` no longer takes `searchParams` — the page reads from
  cookies, not from the URL.
